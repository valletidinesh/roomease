People, Status & Presence

Every rotation module reads two flags per person before doing anything. These two flags together decide who is eligible for a task.

⚡
Status
ValueMeaning
ACTIVEPerson lives here and participates in all rotations.
INACTIVETemporarily removed from all task rotations. Rotation sequences shrink automatically. Reactivation picks up where the sequence left off.
📍
Presence
ValueMeaning
PRESENTAt home today. Included in today's eligible pool.
AWAYOut of town. Skipped for tasks while away. Rotation does not advance for them.
Eligibility rule (used everywhere):
eligible = users WHERE status = ACTIVE AND presence = PRESENT
All modules compute their "who's next" answer from this live filtered set.
Presence isolation — the key rule

If only 2 people are present, the rotation runs between those 2 only. The other 2 are completely invisible to the engine. When they return, they are added back to the eligible pool and the sequence continues — the order of the original sequence is always preserved. No reshuffle happens on return.

01

The Reusable Rotation Engine

Cooking and Trash use one engine. The key insight for cooking: the engine is group-aware. Each unique combination of present people is a distinct group with its own saved cycle state. Trash uses the same engine but without group-splitting (simpler).

Master Order — defined once, never changes

A fixed sequence is set when the app is first configured, e.g. A → B → C → D.
Every group's sequence is derived by filtering the master order to only that group's members.
Example: Master = A B C D → Group {A,D} = A→D  |  Group {A,B,D} = A→B→D  |  Group {B,C} = B→C

Group Identity

How the current group is identified currentGroup = users .filter(u => u.status == ACTIVE && u.presence == PRESENT) .sortBy(masterOrder) // always consistent ordering .map(u => u.id) groupKey = currentGroup.join(",") // e.g. "A,B,D"

State the engine tracks — per group row

group_rotation_state fields (one row per group) groupKey // "A,B,D" — the identity of this group sequence[] // derived from master order: [A, B, D] currentCycleOrder[] // working copy — may be reordered if override happened cycleIndex // pointer into currentCycleOrder lastActualUserId // for consecutive-cook prevention lastActivityAt // for session expiry check

Looking up state when group changes

getGroupState(groupKey) { row = db.find(groupKey) if (!row) { // Brand new group — never cooked together before row = createFreshState(groupKey) // cycleIndex = 0, start from A } // ⚠️ NO time-based reset. State is always resumed as-is. // If ABD was mid-cycle (D is next), D cooks next even after 6 months. // The ONLY reset is the natural cycle reset below. return row }

Normal Turn, Override & Consecutive Check

getAssigned(groupKey) { state = getGroupState(groupKey) return state.currentCycleOrder[state.cycleIndex] } markDone(actualUserId) { state = getGroupState(groupKey) assigned = state.currentCycleOrder[state.cycleIndex] if (actualUserId != assigned) { applyOverride(state, assigned, actualUserId) } checkConsecutive(state, actualUserId) state.cycleIndex++ state.lastActivityAt = NOW() if (state.cycleIndex >= state.currentCycleOrder.size) { state.currentCycleOrder = state.sequence.copy() // reset to A B D state.cycleIndex = 0 } db.save(state) } applyOverride(state, assigned, actual) { state.currentCycleOrder.remove(actual) pos = state.currentCycleOrder.indexOf(assigned) state.currentCycleOrder.insert(pos, actual) // cycleIndex unchanged — now points to actual at assigned's old slot } checkConsecutive(state, actualUser) { if (actualUser == state.lastActualUserId && state.sequence.size > 1) { throw "Cannot cook back-to-back when others are available" } state.lastActualUserId = actualUser }

This cycle becomes A D B for group ABD. After B completes → resets to A B D for next cycle.

When does a group's cycle reset to the beginning?

Only when the cycle completes naturally — every member of the group has cooked once in the current cycle.

If ABD was mid-cycle (A cooked, B cooked, D is next) and the group doesn't form again for 3 months — when they're back together, D still cooks next. The state is always resumed exactly where it was frozen.

cycleIndex = 0 only triggers inside markDone() when cycleIndex reaches currentCycleOrder.size. No time-based reset exists.

03

Trash — Wet & Dry

🗑️
Trash

Split into WET and DRY. 1 person throws at a time. No fixed rotation — whoever has the fewest complete turns goes next. 1 complete turn = 1 WET + 1 DRY thrown.

Complete Turn Definition

complete_turns(person) = min(wet_count, dry_count)

A's wet=3, dry=1 → complete_turns = 1  (the 2 extra wet throws don't count as complete yet)
B's wet=2, dry=2 → complete_turns = 2
C's wet=0, dry=0 → complete_turns = 0 ← goes next for either type

Selection Algorithm

getNextThrower(trashType: WET | DRY) { eligible = users.filter(u => u.status==ACTIVE && u.presence==PRESENT) sorted = eligible.sortBy(u => [ min(u.wetCount, u.dryCount) ASC, // 1. fewest complete turns first u[trashType + "Count"] ASC, // 2. tiebreak: fewest of this specific type u.lastTrashAt ASC // 3. final tiebreak: longest ago ]) return sorted[0] } markTrashDone(userId, trashType) { if (trashType == WET) user.wetCount++ if (trashType == DRY) user.dryCount++ user.lastTrashAt = NOW() }

All Scenarios

ScenarioWho is presentWhat happens

One absent A away, B present B cleans alone. A is skipped. Rotation continues: next turn will be A (when A returns).

Washroom-specific state

washroom_state row washroomNumber : 1 groupOrder : ["A", "B"] // never mutated (no override in washroom) cycleIndex : 0 // 0=A's turn, 1=B's turn lastActualUserId : "A" status : ACTIVE // or PENDING_RESUME when both absent
No override in washroom. The group members clean within their own pair. There is no mechanism to swap in someone from the other group.

Consumables Tracking (Eggs, Bread, Milk…)

🥚
Consumables

No rotation logic. Pure ledger: who bought a batch → how many each person used → split computed at the end.

Two-table model

purchase_entry id item // EGG | BREAD | MILK | OIL | ... totalQty // e.g. 30 (eggs) totalPrice // e.g. 300 (rupees) boughtBy // userId of who bought boughtAt // timestamp status // OPEN | CLOSED
usage_log id purchaseEntryId // FK to entry above userId qty // how many used this time loggedAt // Person picks which entry they're // consuming from when they log usage

Multiple open batches — no merging

Scenario: 10 eggs remain, A buys 30 more

The old entry stays open with its original 30 qty. A creates a new entry for 30 eggs. Both entries are OPEN simultaneously. Users log usage against whichever entry they consumed from. This preserves per-batch accountability (who bought which batch).

EntryBought byTotalUsed so farRemainingStatus
Entry #1 (old)B302010OPEN
Entry #2 (new)A30030OPEN

Split Calculation (computed on demand)

Split for purchase_entry #1 (30 eggs, ₹300, bought by B) usageLogs = [ { userId: A, qty: 8 }, { userId: B, qty: 12 }, { userId: C, qty: 6 }, { userId: D, qty: 4 } ] total used = 30 perPersonShare(userId) = (userQty / totalQty) * totalPrice A → (8/30) * 300 = ₹80 B → (12/30) * 300 = ₹120 // B bought, so B is owed ₹180 (300-120) C → (6/30) * 300 = ₹60 D → (4/30) * 300 = ₹40
Settlement is separate. The consumables module only tracks usage. Who pays whom is handled externally (e.g., Splitwise or a separate expense log). The app stores who bought each batch — that's all that's needed for manual settlement.

Closing a batch

When a user marks an entry as CLOSED (all consumed or entry no longer tracked), the final split snapshot is computed and stored. No further usage can be logged against a closed entry.

Water Can Fetching

💧
Water Cans

2 people go together each time. Uses load-balanced pairing — not a fixed rotation list. Always picks the 2 eligible people who have fetched the fewest times.

Why not a fixed sequence?

A fixed pair sequence (A+B, C+D, A+C …) breaks fairness the moment someone's presence changes mid-week. Load-balanced selection fixes this: whoever has gone the fewest times always goes next, regardless of when they joined or returned.

The algorithm

getNextWaterPair() { eligible = users.filter(u => u.status==ACTIVE && u.presence==PRESENT) if (eligible.size < 2) throw "Need at least 2 people to fetch water" sorted = eligible.sortBy(u => [u.waterFetchCount ASC, u.lastFetchedAt ASC]) // primary sort: lowest fetch count first // tiebreak sort: who went longest ago comes first return [sorted[0], sorted[1]] } markWaterDone() { pair = getNextWaterPair() pair.forEach(u => { u.waterFetchCount++ u.lastFetchedAt = NOW() }) }

Scenario: 3 people present (A, B, C)

Fetch #Counts beforePair chosenCounts after
1A=0, B=0, C=0A + B (tied, alphabetical tiebreak)A=1, B=1, C=0
2A=1, B=1, C=0C + A (C=0 first, A tiebreaks B by time)A=2, B=1, C=1
3A=2, B=1, C=1B + C (lowest counts)A=2, B=2, C=2
4A=2, B=2, C=2All tied → A + B againA=3, B=3, C=2
D returns from a trip mid-cycle: D's count = 0, everyone else = 2+. D immediately goes next regardless of whose "turn" it was. The count-based system self-corrects instantly. No manual reset needed.

What triggers a fetch?

Manual trigger — any user taps "Get water now" → system shows the next pair → they go → either of them marks it done → counts increment → system shows the new next pair automatically.

Water state stored per user

additional columns on users table waterFetchCount INT DEFAULT 0 lastFetchedAt TIMESTAMP NULL

Buy List

🛒
Buy List

A shared list anyone can add to. When someone goes shopping, they check the list, buy items, and mark them bought. Optionally creates a purchase entry in consumables.

Lifecycle of a buy list item

PENDING → Item added by any user. Visible to all.
BOUGHT → Marked bought by whoever shopped. Records boughtBy + boughtAt.
→ Purchase Entry → Optional: auto-creates a consumable entry when marked bought (with qty + price filled in).
buy_list row id itemName : "Eggs" addedBy : userId addedAt : timestamp status : PENDING | BOUGHT boughtBy : userId (null until bought) boughtAt : timestamp (null until bought)

Edge Cases Summary

ModuleEdge CaseHandled How
Cooking Only 1 person present, cooks 3 days in a row Single-person group (e.g., "A"). Consecutive check bypassed when group size == 1. No error.
Cooking Group changes mid-day (B marks present at 3pm) Group key is evaluated at cook time (when someone taps "mark done"). If B is present when done is marked, B is in the group.
Cooking D overrides B in ABD, then D tries to cook again immediately Consecutive check on ABD group state: lastActualUserId == D, group size > 1 → rejected.
Cooking ABD hasn't cooked together for 3 months, D was next State is resumed as-is. D cooks next. No time-based reset. Cycle only resets when it completes naturally.
Washroom Both group members absent Status → PENDING_RESUME. Washroom uncleaned. No cross-group interference. Resumes when either returns.
Washroom New person moves in (5th roommate) Admin assigns them to a group. Appended to that group's rotation array.
Water D returns with count = 0, others at count = 5 D is immediately selected for next pair. Load balancing self-corrects without any manual reset.
Water Only 1 person present API returns error: minimum 2 required to fetch water. UI shows "waiting for someone to be present".
Consumables New batch bought before old batch finished Separate purchase entries created. Both remain OPEN. User chooses which entry to log usage against.
Consumables More usage logged than totalQty API validation: SUM(usage_log.qty) must not exceed purchase_entry.totalQty for OPEN entries. Return 400 if exceeded.
General All users INACTIVE simultaneously All tasks show "No eligible users." No rotation state mutates. Resumes when someone is reactivated.

Build Phases

Phase 1
Core Engine + Cooking

User model (status + presence). Generic RotationEngine class. Cooking rotation fully working end-to-end: override, consecutive check, cycle reset. REST API for cooking. Basic dashboard endpoint.

users table rotation_config rotation_state task_history
Phase 2
Trash + Washroom + Water

Reuse RotationEngine for trash (literally plug in with taskType=TRASH). Washroom variant with group isolation and PENDING_RESUME status. Water load-balanced pair selection using fetch counts.

washroom_groups washroom_state water_fetch_count on users
Phase 3
Consumables + Buy List

Purchase entries, usage logging, and computed splits. Buy list with PENDING → BOUGHT lifecycle. Optional auto-creation of purchase entry when buy list item is marked bought.

purchase_entry usage_log buy_list
Phase 4
Dashboard + Polish

Unified /dashboard endpoint. History views per task. Split summary per consumable entry. Presence management UI. Admin tools for group assignment and reactivation. Mobile-first frontend.

/dashboard Frontend (React Native / Flutter / PWA)

Recommended Stack

Backend

Spring Boot (Java/Kotlin) — clean service/repository separation. RotationEngine becomes a @Service bean. Each task type has its own @Service that delegates to the engine.

Or: Node.js + Express + TypeScript if you prefer JS ecosystem.

Database

PostgreSQL — JSON columns for rotation arrays, solid FK support, easy migration with Flyway / Liquibase.

SQLite is fine for a local prototype — swap to Postgres for multi-device access.

Frontend

React Native for a shared iOS/Android app. Or a PWA (React + Vite) — everyone opens it in browser, no app store needed.

A shared dashboard everyone can open is sufficient — no push notifications for MVP.
